# 高级工程参考

本文保存偏未来阶段的高级工程约束。只有当项目进入对应实现阶段时，才应把这些条目提升到正式架构文档或里程碑中。

当里程碑涉及 ID、幂等、并发、状态机、时间、搜索、第三方集成、离线同步、兼容性、埋点、成本或许可证时，应先检查本清单。

## 1. ID 设计

不要随便用自增 ID 暴露给前端。

要考虑：

```text
用户 ID 怎么生成
项目 ID 怎么生成
订单 ID 怎么生成
是否暴露数据库主键
是否需要 publicId / internalId 分离
ID 是否可被枚举
```

推荐前端看到的是：

```text
user_abc123
proj_abc123
file_abc123
```

而不是：

```text
1
2
3
4
```

否则容易被猜 ID，增加越权风险。

---

## 2. 幂等性

很多接口不能简单地“请求一次就执行一次”。

例如：

```text
创建订单
支付回调
发送邮件
上传文件
创建任务
扣额度
邀请用户
```

如果用户网络抖动重复点击，或者前端重试，后端不能重复创建两条数据。

要考虑：

```text
Idempotency-Key
唯一约束
重复请求检测
任务状态机
支付回调去重
```

---

## 3. 并发和事务

后端不是单用户顺序执行。多个请求可能同时改同一份数据。

要考虑：

```text
两个人同时编辑同一个项目
两个请求同时扣额度
重复提交表单
同时邀请同一个成员
同时删除和更新同一资源
```

需要设计：

```text
数据库事务
唯一索引
乐观锁 version
行级锁
状态机校验
```

---

## 4. 状态机

很多业务不是简单的 true/false。

例如任务可能有：

```text
pending
running
succeeded
failed
cancelled
expired
```

订单可能有：

```text
created
paid
refunded
cancelled
failed
```

需要限制非法跳转：

```text
pending -> running 可以
running -> succeeded 可以
succeeded -> running 不应该可以
cancelled -> succeeded 不应该可以
```

否则后期数据会乱。

---

## 5. 时间和时区

这个非常容易被低估。

统一原则：

```text
数据库存 UTC
API 返回 ISO 8601
前端按用户本地时区展示
不要用字符串乱拼时间
定时任务明确时区
```

特别是：

```text
订阅到期时间
日志时间
任务开始/结束时间
邮件发送时间
每日额度刷新时间
```

都要清楚。

---

## 6. 搜索系统

一开始可以数据库 `LIKE`，但后期如果有大量项目、文件、消息、文档，可能要考虑搜索。

要想：

```text
搜索哪些字段
是否支持模糊搜索
是否支持中文分词
是否支持排序
是否支持权限过滤
搜索结果是否实时
```

不要让用户能搜到自己无权访问的数据。

---

## 7. Webhook / 第三方回调

如果以后接支付、GitHub、Slack、邮箱、AI 服务，可能会有 webhook。

Webhook 要考虑：

```text
签名校验
重复回调
乱序回调
失败重试
事件日志
回调版本
```

尤其支付类 webhook，必须严谨。

---

## 8. 邮件投递

注册、验证码、邀请、账单、通知都可能发邮件。

要考虑：

```text
邮件模板
退信处理
发送频率限制
验证码过期
邀请链接过期
邮件队列
unsubscribe
发件域名配置
```

不要在 HTTP 请求里同步等邮件发完。

---

## 9. 用量额度系统

如果产品有 AI、存储、导出、团队协作，迟早会有额度。

要提前想：

```text
免费用户每天多少次
付费用户多少额度
团队额度还是个人额度
额度什么时候刷新
失败请求扣不扣额度
重试请求怎么处理
```

额度系统一定要和幂等性、事务一起设计。

---

## 10. 数据导入导出

很多产品后期都会需要：

```text
导入 CSV
导出 CSV
导出 JSON
导出 PDF
批量上传
批量下载
迁移数据
```

要考虑：

```text
大文件怎么处理
失败行怎么提示
导出是否异步
导出文件保存多久
导出权限谁有
```

---

## 11. 数据修复工具

线上数据一定会出现异常。

需要内部工具或脚本：

```text
修复用户状态
重算额度
重跑失败任务
重新发送通知
恢复误删数据
检查数据一致性
```

不要每次都手动进数据库改。

---

## 12. 多租户设计

如果以后有团队、组织、workspace，就要提前考虑多租户。

核心是：

```text
数据属于哪个 workspace
用户在 workspace 里是什么角色
API 查询是否永远带 workspaceId
管理员能不能跨 workspace
邀请成员如何生效
```

很多 B2B 产品后期最难补的就是多租户。

---

## 13. 缓存失效策略

用 Redis 或前端缓存时，不只是“加缓存”。

要想：

```text
缓存 key 怎么命名
缓存多久过期
数据更新后怎么失效
权限变化后缓存怎么清
用户退出后缓存怎么清
```

缓存最怕的是：数据改了，但用户看到旧数据，甚至看到不该看的数据。

---

## 14. 前端权限和后端权限分离

前端可以控制展示：

```text
隐藏按钮
禁用入口
不显示菜单
```

但真正权限必须在后端判断。

也就是：

```text
前端权限 = 用户体验
后端权限 = 安全边界
```

不要只靠前端判断。

---

## 15. 桌面端安装包和签名

如果项目包含 Tauri 桌面端，还要考虑：

```text
macOS code signing
Windows code signing
自动更新签名
安装包渠道
崩溃日志
旧版本兼容
强制升级策略
```

桌面端不是打包完就结束，分发和更新也要维护。

---

## 16. Offline / Sync 策略

如果 Tauri 客户端支持离线使用，就要设计同步。

要考虑：

```text
离线时能做什么
本地数据怎么存
重新联网怎么同步
冲突怎么解决
谁覆盖谁
失败怎么提示
```

冲突策略可以是：

```text
最后写入者获胜
手动冲突解决
服务端为准
客户端为准
按字段合并
```

这块如果不做，就明确第一版“不支持离线编辑”。

---

## 17. 兼容性矩阵

需要明确支持哪些环境。

例如：

```text
Web 支持哪些浏览器
Tauri 支持哪些系统版本
后端支持哪些部署环境
数据库版本
Node 版本
Rust 版本
```

否则用户环境一多，问题很难排。

---

## 18. 事件埋点字典

如果要分析用户行为，不要随便埋点。

要维护事件字典：

```text
user_signed_up
project_created
file_uploaded
export_started
export_completed
payment_started
payment_succeeded
```

每个事件要有固定属性：

```text
userId
workspaceId
timestamp
source
plan
result
```

否则数据分析会非常乱。

---

## 19. 成本控制

服务器、数据库、对象存储、邮件、AI API 都有成本。

要提前监控：

```text
每个用户成本
每个 workspace 成本
AI 请求成本
对象存储成本
数据库成本
日志成本
带宽成本
```

尤其如果产品涉及 AI，一定要有限流、额度和成本报警。

---

## 20. 开源许可证合规

如果使用了第三方库、图标、字体、模板，要确认许可证。

要记录：

```text
库名
版本
许可证
是否允许商用
是否需要署名
是否能二次分发
```

特别是桌面客户端会把依赖一起分发，更要注意。

---

# 重点检查项

优先补齐以下 8 项：

```text
1. ID 设计
2. 幂等性
3. 并发事务
4. 状态机
5. 时间/时区
6. 多租户/workspace
7. 缓存失效
8. 桌面端签名和自动更新
```

完成这些约束后，项目 checklist 会从：

```text
怎么开发
```

扩展到了：

```text
怎么上线
怎么协作
怎么排查
怎么扩展
怎么收费
怎么保护数据
怎么长期维护
```

后续应根据具体产品类型继续补充。AI 工具、协作工具、文件工具、开发者工具和 SaaS 后台的重点不同，遗漏项也会不同。
