# 产品治理参考

本文保存产品治理和生命周期约束。已经接受的正式规则位于 `docs/product/*`、`docs/operations/security.md` 和里程碑文档。

除了工程架构、API 对接、部署、运维和安全之外，产品治理还需要下面这些边界。

## 1. 产品需求边界

技术方案之前，要先明确：

```text
这个产品解决什么问题
核心用户是谁
第一版必须有什么
哪些功能暂时不做
成功指标是什么
```

否则很容易变成“架构很完整，但产品范围失控”。

建议维护一份：

```text
MVP 功能清单
非目标功能清单
用户角色
核心流程图
优先级
```

---

## 2. 用户角色和后台管理

很多产品一开始只考虑普通用户，忘了管理端。

需要想：

```text
有没有管理员
管理员能看什么
能不能封禁用户
能不能查看日志
能不能手动处理订单/文件/异常
能不能配置系统参数
```

如果有运营或客服，可能还需要：

```text
Admin Dashboard
用户查询
数据修复入口
工单处理
审计记录
```

---

## 3. 审计日志

普通日志是给开发排查的，**审计日志**是记录关键行为的。

例如：

```text
谁登录了
谁删除了项目
谁导出了数据
谁修改了权限
谁邀请/移除了成员
谁改了计费信息
```

这类日志后期很重要，尤其是团队协作、企业用户、付费系统。

---

## 4. 数据生命周期

需要提前想清楚数据怎么“出生、变化、删除”。

包括：

```text
用户删除账号后数据怎么办
项目删除是软删除还是硬删除
文件是否可恢复
回收站保留多久
日志保留多久
备份保留多久
导出数据格式是什么
```

特别是删除操作，不要一开始就随便 `DELETE FROM`。很多时候更适合：

```text
soft delete
deleted_at
restore
permanent delete
```

---

## 5. 隐私与合规

不一定一开始就做复杂合规，但至少要知道自己存了什么数据。

维护一张表：

```text
收集哪些用户数据
为什么收集
存在哪里
保留多久
谁能访问
是否会传给第三方
用户能否导出/删除
```

如果有登录、文件、聊天、AI、支付、团队协作，这部分要更重视。

---

## 6. 可访问性

前端不要只考虑“好看”，还要考虑可用性。

至少考虑：

```text
键盘可操作
焦点状态清晰
颜色对比度足够
表单错误提示明确
按钮有可读 label
图标不能单独承担含义
```

尤其桌面客户端，很多用户会依赖快捷键和键盘导航。

---

## 7. 国际化和本地化

即使第一版只做中文，也建议不要把所有文案硬编码死。

可以提前规划：

```text
zh-CN
en-US
日期格式
时间格式
数字格式
货币格式
错误文案
空状态文案
```

如果未来要出海外版，早期不做 i18n 结构，后面会很痛苦。

---

## 8. 设计走查和 UI 回归

前文提到风格维护，但还要考虑怎么防止页面越改越丑。

可以做：

```text
组件预览页
设计系统页面
截图回归测试，可选
UI checklist
页面验收标准
```

每次新增页面都检查：

```text
颜色是否用了 token
间距是否统一
按钮状态是否完整
loading/empty/error 是否有
暗色模式是否正常
```

---

## 9. 性能预算

不要等卡了再优化。

提前定一些预算：

```text
首屏加载时间
接口响应时间
列表最大渲染数量
图片大小限制
bundle size
数据库查询耗时
后台任务超时时间
```

前端要考虑：

```text
代码分包
懒加载
虚拟列表
图片压缩
缓存策略
```

后端要考虑：

```text
索引
慢查询
连接池
缓存
分页
批量操作
任务队列
```

---

## 10. 测试数据和种子数据

开发时不要每个人手动造数据。

建议准备：

```text
seed users
seed projects
seed permissions
seed files
测试账号
本地 mock 数据
staging demo 数据
```

这样前端、后端、测试、演示都会顺很多。

---

## 11. Feature Flag

有些功能不应该一上线就对所有人开放。

可以考虑功能开关：

```text
新功能灰度
内部测试
按用户开启
按组织开启
快速关闭问题功能
```

例如：

```text
enableNewEditor
enableAiFeature
enableDesktopSync
enableBetaBilling
```

这对长期迭代很有用。

---

## 12. 通知系统

很多产品后期都需要通知：

```text
站内通知
邮件通知
桌面通知
任务完成通知
权限变更通知
失败提醒
账单提醒
```

要提前想：

```text
通知类型
通知渠道
用户是否能关闭
是否需要已读/未读
是否需要通知历史
```

Tauri 端还可能涉及系统通知权限。

---

## 13. 支付和订阅

如果未来商业化，要提前考虑：

```text
免费版/付费版
套餐限制
用量统计
账单状态
试用期
取消订阅
发票
支付失败处理
额度超限
```

即使第一版不做，也要避免后端数据模型完全无法支持。

---

## 14. 用量统计和产品分析

需要知道用户怎么用产品。

可以统计：

```text
注册转化
活跃用户
功能使用次数
关键流程完成率
错误率
留存
付费转化
```

但要注意隐私，不要记录敏感内容。

---

## 15. 客服和反馈入口

用户遇到问题时，要有路径反馈。

可以准备：

```text
反馈入口
错误上报
复制诊断信息
上传日志，可选
联系支持
问题分类
```

桌面客户端尤其建议有：

```text
导出诊断包
查看日志路径
复制 app version / system info
```

---

## 16. 版本管理

前端、后端、桌面端都要有版本策略。

需要明确：

```text
后端 API 版本
Web 前端版本
Tauri 客户端版本
数据库 schema 版本
配置版本
客户端最低支持版本
```

尤其桌面端很重要，因为用户可能长期不更新。

需要考虑：

```text
旧客户端还能不能访问新后端
什么时候强制更新
API 是否兼容旧版本
自动更新失败怎么办
```

---

## 17. 依赖升级策略

前端和 Rust 依赖都会不断变化。

需要定：

```text
多久检查一次依赖
安全更新怎么处理
大版本升级谁负责
锁文件怎么维护
废弃库怎么替换
```

不要长期不升级，也不要随便全量升级。

---

## 18. 文档体系

不是写很多文档，而是写必要文档。

建议至少有：

```text
README：怎么启动项目
ARCHITECTURE：整体架构
API：接口规范
CONTRIBUTING：代码规范
DEPLOYMENT：部署流程
SECURITY：安全注意事项
RUNBOOK：线上故障处理
```

团队变大或者过几个月回来维护时，这些很有用。

---

## 19. Runbook / 故障处理手册

上线后一定会出问题。要提前写：

```text
服务挂了怎么查
数据库连不上怎么查
接口 500 怎么查
队列堆积怎么查
磁盘满了怎么处理
如何回滚
如何恢复备份
如何临时关闭某功能
```

这类文档很少有人一开始写，但出事时价值极高。

---

## 20. 法务和产品条款

如果是真正上线产品，至少要准备：

```text
用户协议
隐私政策
免责声明
数据删除说明
第三方服务说明
开源许可证合规
```

如果使用了开源库、图标、字体、模型、第三方 API，也要确认授权。

---

# 最终补全版清单

当前应当把整个项目拆成这些维度来看：

```text
产品：
MVP、用户角色、核心流程、商业模式

前端：
UI 风格、组件系统、状态管理、可访问性、国际化

客户端：
Tauri、本地缓存、自动更新、系统权限、离线状态

后端：
Rust API、分层架构、认证鉴权、任务队列、错误码

数据：
PostgreSQL、Redis、对象存储、migration、备份恢复

对接：
API contract、类型生成、错误规范、分页规范、版本兼容

安全：
权限、限流、密钥、审计、上传安全、依赖安全

运维：
CI/CD、日志、监控、告警、回滚、健康检查

增长/运营：
Admin、通知、反馈、用量统计、支付订阅

长期维护：
文档、runbook、版本策略、依赖升级、合规
```

## 优先补齐项

按优先级，后续重点考虑这 8 个：

```text
1. 用户角色和权限模型
2. API contract 和错误码体系
3. 数据库 schema + migration
4. 登录认证和 token/session 策略
5. 日志/requestId/监控
6. CI/CD 和部署环境
7. 备份恢复
8. Tauri 客户端版本更新策略
```

项目已经覆盖“怎么把项目写出来”，后续还要覆盖 **怎么让它上线、可查、可控、可恢复、可长期迭代**。
